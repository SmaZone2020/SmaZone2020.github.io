## 一、恶意模组分析

### 概述

**重要说明**：反作弊系统本身是合法的安全工具，许多开源反作弊项目（如Minecraft的各类反作弊插件）为维护游戏公平性做出了重要贡献。本分析**不针对反作弊技术本身**，而是针对**利用开源反作弊框架嵌入恶意代码**的行为。

`[YJFS]14.jar`、`[YJFS]15.jar` 和 `[YJFS]16.jar` 是利用开源反作弊框架，**在合法反作弊功能中嵌入恶意代码**的间谍软件/远控木马 (RAT)。这种方式使得恶意代码更容易被忽视，因为：
- 用户会认为反作弊模组是"正常"的
- 恶意功能可以伪装成反作弊功能
- 开源代码提供了合法的外壳，便于隐藏恶意行为

### 开源反作弊被滥用的风险

开源反作弊项目本身是透明、可审计的，但当被恶意修改后：
1. **代码混淆** - 使用Allatori、Radon等混淆器隐藏恶意代码
2. **功能伪装** - 将远程命令执行伪装成"游戏检测"
3. **通信隐藏** - 将C2通信伪装成"心跳验证"
4. **监控伪装** - 将全屏截图伪装成"画面检测"

这提醒我们：**使用第三方反作弊模组时，必须确保其来源可信、代码未被篡改。**

### [YJFS]14.jar — "MaoJiNianCAC"

#### 基本信息

- **modid**: `maojiniancac`
- **伪装名称**: MaoJiNianCAC (猫几年反作弊)
- **声称作者**: maojinian
- **Minecraft版本**: 1.6.4
- **混淆器**: Allatori Demo (字符串加密 + 控制流混淆)
- **恶意代码包**: `l/l/l/` (38个class文件)
- **捆绑库**: JNA, Netty, Apache Commons Codec, Gson

#### 后门架构

```
入口点加载链:
┌─────────────────────────────────────────────────────┐
│ l/l/l/l/?c (IFMLLoadingPlugin)                      │ ← Forge最早加载
│   └─ 注册 l/l/l/l/?e 为 ASM ClassTransformer       │
├─────────────────────────────────────────────────────┤
│ l/l/l/?d (@Mod主类, IPacketHandler)                  │ ← FML Mod入口
│   ├─ preload() → l/l/l/?H 初始化配置+文件           │
│   ├─ init() → 启动 l/l/l/?d$?A 反调试线程          │
│   ├─ init() → 注册5个网络频道接收服务端指令          │
│   ├─ init() → 注册 MinecraftForge EVENT_BUS         │
│   └─ onPacketData() → 分发服务端下发的命令           │
└─────────────────────────────────────────────────────┘
```

#### 命令下发与执行流程

**下发方式**: 通过 Forge NetworkRegistry 注册自定义频道，服务端发送 Packet (类型 `ea`)，客户端 `onPacketData()` 方法接收。

**指令分发** (`l/l/l/?d.onPacketData`):

1. **频道1** → `l/l/l/a/?L.ALLATORIxDEMOx?(byte[])` — 接收二进制文件分片并写入磁盘
2. **频道2** → `l/l/l/l/?D.ALLATORIxDEMOx?(String)` — 执行JSON命令（主命令处理器）
3. **频道3** → `l/l/l/a/?L.ALLATORIxDEMOx?(String)` — 文件完整性校验
4. **频道4/5** → 其他控制指令

#### 恶意功能详解

##### 1. 远程命令执行 — `l/l/l/l/?g`

该类接收一段字符串（VBScript代码），创建临时文件写入脚本内容，然后通过 `Runtime.exec()` 执行该临时VBS文件，读取执行输出后删除临时文件。

```
流程: 收到字符串 → 写入 .vbs 临时文件 → Runtime.exec("wscript 临时文件路径") → 读取输出 → 删除临时文件
```

这意味着服务端可以远程在玩家电脑上执行**任意VBScript代码**。

##### 2. 信息收集与上报 — `l/l/l/l/?l`

主数据收集和上报类。当收到服务端的"配置更新"指令时，执行以下操作：

- 解析JSON配置(含Display.setTitle设置窗口标题掩饰)
- 收集客户端RSA加密密钥 (`l/l/l/l/?K`)
- 收集玩家列表 (黑白名单)
- 收集已加载mod列表 (`l/l/l/l/?F`)
- 收集注册事件监听器列表 (`l/l/l/?f`)
- 收集 ClassLoader 中所有类的MD5哈希 (`l/l/l/l/?e`)
- 计算特定文件的MD5作为机器指纹 (`DigestUtils.md2Hex`)
- 将所有数据打包为 JSON → 用RSA加密 → Base64编码 → 构建URL上报

##### 3. RSA加密通信 — `l/l/l/l/?j`

使用硬编码的RSA公钥对上报数据加密：

- `KeyFactory.getInstance("RSA")`
- `X509EncodedKeySpec` 加载公钥
- `Cipher.getInstance("RSA")` 加密
- 确保即使截获数据包也无法解密内容

##### 4. ClassTransformer间谍 — `l/l/l/l/?e`

实现 `IClassTransformer` 接口，注入到Minecraft类加载管线中：

- 使用反射获取 `URLClassLoader.ucp` 字段
- 遍历所有已加载的 URL (jar路径)
- 对每个jar计算 MD5 指纹
- 收集所有非标准库的类名(截断到一定长度)
- 数据存入 `ConcurrentSet` 供上报

##### 5. 文件下载执行 — `l/l/l/a/?L`

支持服务端分片下载二进制文件到客户端：

- 接收带有文件名+路径的二进制数据
- 自动创建目录
- 拼装分片后写入 `BufferedOutputStream`
- 支持在指定路径创建任意文件（可能用于投放后续payload）

##### 6. 反调试与自毁 — `l/l/l/?d$?A`

内部线程类，通过 `Thread.sleep` + `System.nanoTime` 差值检测调试器：

- 计算 sleep 前后时间差
- 如果时间差异常(被断点暂停)则进入死循环
- 多处 `Thread.stop()` / `Thread.interrupt()` 用于中止可疑线程

##### 7. Mod列表扫描 — `l/l/l/?H`

在 `FMLPreInitializationEvent` 阶段：

- 扫描 `.minecraft/mods` 目录下所有子目录名
- 拼接成字符串作为"已安装mod"报告
- 创建配置文件(写入默认值如果首次运行)
- 读取保存的配置

##### 8. 键位监控 — `l/l/l/l/?k`

注册 `KeyBindingRegistry$KeyHandler`，监控特定组合键：

- 检测两个键同时按下
- 将按键代码通过网络包发送到服务端
- 服务端可能用此收集玩家行为数据

### [YJFS]15.jar — 屏幕截图模块

#### 基本信息

- **文件名**: `[YJFS]15.jar`
- **混淆器**: 未知（类名已混淆）
- **主要功能**: 全屏幕截图并发送到服务端

#### 恶意代码位置

**类路径**: `mcwk.Mana.l11Ii1IIIiIII1IIl11l11Ii1IIIiIII1IIl11i1I1i1liili1I1lil1iIi`

**关键代码** (第28-29行):

```java
Dimension var1 = Toolkit.getDefaultToolkit().getScreenSize();
BufferedImage var2/* $VF was: 10x */ = new Robot().createScreenCapture(new Rectangle(0, 0, (int)var1.getWidth(), (int)var1.getHeight()));
```

#### 代码分析

| 调用 | 作用 | 问题 |
|------|------|------|
| `Toolkit.getDefaultToolkit().getScreenSize()` | 获取**整个屏幕**的尺寸 | 获取的是屏幕分辨率，不是游戏窗口 |
| `new Robot()` | 创建屏幕截图工具 | Java AWT Robot类 |
| `createScreenCapture(new Rectangle(0, 0, width, height))` | 截取**整个屏幕** | 从(0,0)开始，覆盖整个屏幕 |

#### 与隐私政策的矛盾

**隐私政策声称**:
> "采集范围仅限《易境逢生》游戏客户端窗口内的游戏画面，绝不截取用户电脑桌面、其他软件窗口、浏览器界面、社交软件、支付界面"

**实际情况**:
- `getScreenSize()` 获取的是整个屏幕的尺寸
- `createScreenCapture(new Rectangle(0, 0, width, height))` 截取的是**整个屏幕**
- 用户在截图时打开的**任何窗口**都会被截取，包括：
  - 桌面
  - 浏览器（可能包含登录的账号密码）
  - 社交软件（微信、QQ聊天窗口）
  - 支付界面（支付宝、微信支付）
  - 本地文件夹
  - 摄像头画面
  - 其他任何可见内容

#### 截图范围对比

| 场景 | 隐私政策声称 | 实际截取 |
|------|--------------|----------|
| 游戏全屏 | 游戏画面 | 整个屏幕 |
| 游戏窗口化 | 仅游戏窗口 | 整个屏幕 |
| 切出游戏 | 不截取 | 整个屏幕 |
| 多显示器 | 未说明 | 可能截取所有显示器 |

#### 恶意功能详解

##### 1. 远程命令执行 — `mod_CanYiAntiCheat.boom()`

直接调用 `Runtime.getRuntime().exec(String[])` 执行系统命令：

- 构造命令数组执行任意程序
- 读取命令输出 (Scanner + getInputStream)
- 返回执行结果

##### 2. 屏幕截图 — `mod_CanYiAntiCheat` 内部方法

使用 OpenGL 截取游戏画面：

- `GL11.glPixelStorei` + `GL11.glReadPixels` 读取帧缓冲
- 创建 `BufferedImage` 存储像素
- `ImageIO.write(image, "png", ByteArrayOutputStream)` 编码
- 将截图字节数组通过网络发送到服务端

##### 3. 独立Socket后门 — `SocketClient.java`

绕过Minecraft协议的原始TCP连接：

- `new Socket(ip, port)` 直接连接C2服务器
- 独立线程发送数据 (`OutputStream.write(byte[])`)
- 不依赖Minecraft在线状态

##### 4. 进程扫描 — `DllCheck.java`

定期扫描系统进程列表：

- `Runtime.exec()` 执行系统命令列出进程
  - 获取 `%APPDATA%` 等环境变量构造路径
  - 组合路径执行 tasklist 或 wmic 命令
- `BufferedReader` 逐行读取输出
- 对比 `HashMap` 中的关键词检测特定软件
- 发现目标进程时触发 `boom` 操作

##### 5. 数据处理/文件窃取 — `DataHandler.java`

处理从服务端下发的数据操作指令：

- 解析指令字符串判断操作类型
- **文件下载**: 接收 ip+port+data，通过 SocketClient 发送数据
- **目录遍历**: 使用 `File.listFiles(FileFilter)` + 队列遍历指定目录
- **文件搜索**: 正则匹配文件名 (`matches`)
- **数据压缩**: `GZIPOutputStream` 压缩后发送
- **QQ目录扫描**: 专门搜索QQ号相关目录
- 结果通过 `PacketDispatcher.sendPacketToServer` 或 SocketClient 回传

##### 6. AES加密通信 — 内置加密方法

使用 AES/CBC 加密通信数据：

- `Cipher.getInstance("AES/CBC/PKCS5Padding")`(从混淆字符串还原)
- `SecretKeySpec` + `IvParameterSpec` 初始化
- `sun.misc.BASE64Encoder` / `BASE64Decoder` 编解码
- 硬编码密钥和IV向量

##### 7. 定时心跳 — TimerTask

通过 `Timer.scheduleAtFixedRate` 注册定时任务：

- 内部匿名 `TimerTask` 类
- 定期获取玩家名/世界信息
- 构造心跳包发送到服务端
- 保持与C2的持续通信

## 二、哔哩哔哩验证机制分析

### 实现方式

服务器声称的"哔哩哔哩验证"**并非通过哔哩哔哩官方开放平台对接**，而是：

1. 服务器通过网页端接口生成登录二维码
2. 玩家使用手机端哔哩哔哩APP扫码授权
3. 服务器获取用户的**网页端登录凭证**（而非仅获取用户身份信息）

### 与官方API验证的区别

| 对比项 | 官方OAuth2.0 | 服务器实现 |
|--------|--------------|------------|
| 对接方式 | 哔哩哔哩开放平台API | 网页端登录二维码 |
| 授权范围 | 仅获取用户身份信息（uid、昵称） | 获取完整登录凭证 |
| 权限控制 | 申请特定权限，用户可选择拒绝 | 无法拒绝，否则无法登录 |
| 透明度 | 明确告知授权范围 | 未说明实际获取的权限 |
| 安全性 | 官方监管，有安全机制 | 第三方控制，风险未知 |

### 风险分析

由于服务器获取的是**网页端登录凭证**而非OAuth2.0授权，理论上服务器管理员拥有：

#### 账户访问权限
- 登录玩家的哔哩哔哩账户
- 获取用户的完整会话信息
- 模拟用户的所有网页端操作

#### 资产盗用风险
- **硬币** - 可用于投币给视频
- **B币** - 可用于购买大会员、打赏等
- **收藏夹** - 可查看或删除用户收藏
- **历史记录** - 可查看用户浏览历史

#### 非法操作风险
- 在视频评论区发布广告
- 在弹幕中发布不当内容
- 刷赞、刷评论
- 关注/取关其他用户
- 修改用户个人信息

#### 隐私泄露风险
- 获取用户的私信内容
- 获取用户的观看历史
- 获取用户的搜索记录
- 获取用户的地理位置信息

## 三、支付系统与运营资质分析

### 域名备案情况

| 域名 | 备案状态 | 说明 |

### 隐私政策本身的结构性漏洞

即使暂时搁置实际行为与政策的矛盾，隐私政策文本本身也存在严重的结构性缺陷：

- 如果发生数据泄露，如何通知用户？
- 用户如何查询自己被收集了哪些数据？

**4. 缺少未成年人保护**

- 未成年玩家的特殊保护措施
- 是否需要监护人同意
